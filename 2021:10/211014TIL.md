# 📖 학습 내용
**IOS가 다른 플랫폼과 차별되는 세가지 기본 테마**

- 명확성(Clarity) - 텍스트 가독성, 앱의 기능을 알 수 있는 명확한 아이콘, 중요 요소 강조, 기능을 앞세운 디자인

- 존중(Deference) - 컨텐츠 이외의 요소가 더 부각되지 않게함. 밝은 인터페이스 유지, 베젤, 그라데이션, 그림자 사용 최소화

- 깊이(Depth) - 뚜렷한 시각적 레이어와 사실적인 모션(제스처)은 계층구조를 사용자에게 전달하고, 앱에 활력을 부여하며 이해를 용이하게 함

**UIKit에서 제공하는 인터페이스 요소는 세가지의 주요 범주로 나뉨**

- Bars - 사용자에게 네이게이션을 제공하는 것, 네비게이션은 작업을 초기화하고, 정보 전달을 위한 버튼과 기타 요소가 포함됨

- Views -사용자가 볼 주요 컨텐츠가 포함된 것, 주요 컨텐츠는 텍스트나 그래픽, 애니메이션 등 기타 interaction 요소

- Controls -  작업 시작과 정보 전달

테이블 뷰 - 리스트형식을 보여주는 뷰 - 리스트의 한줄한줄을 raw(열)라고 부름 - 각 raw는 TableViewCell

텍스트 필드 - Single Line - 가로 길이가 넘어가도 줄바꿈이 되지않고 계속 입력됨

텍스트 뷰 - Mutli Line  - 가로 길이가 넘어가면 줄바꿈

**모달과 네비게이션은 정보의 흐름의 차이**

- 네비게이션은 정보의 깊이와 흐름을 가지는 구조
- 모달은 정보의 흐름에서 벗어나 전환해주는 것 - 작성 폼, alert, action sheet - 정보의 흐름과는 별개 (temporary stage)

**Example**

- hierarchical : 설정
- flat : 인스타그램이나, 유튜브같이 탭마다 구조를 다르게 가지고, 탭마다 다른 계층(깊이)를 가짐



# 🔍 의문
struct가 각각 구현이 되어있을 때, 동일한 기능이 두 struct에 필요하다면 함수명이 같게 구현을 해도 되는가?에 의문을 가졌다.

- struct, 즉 구조체가 다르기때문에 명확히 동일한 기능을 하는 함수라면, 함수명이 같아도 괜찮다. 
- 애플에서도 delegation 등에 대한 함수가 파라미터만 다르고 동일한 함수명인 경우가 많다.
- 함수명이 같아도 다른 구조체 내부에 선언된 메소드는 각각의 인스턴스 생성 후 dot을 통해 접근할 수 있으므로, 함수명이 같아도 그 기능을 가진 객체(struct)가 다르므로 상관이 없음

그럼, 이렇게 재사용할 수 있는 함수는 최대한 재사용을 하는 것이 좋은가?

- 맞음, 함수는 애초에 많이 쓰일 것 같은 코드를 묶어 하나의 기능, 즉 함수로 묶는 것이다. 묶어서 쉽게 재사용도 하고, 재사용을 함으로써 코드수도 줄게되어 가독성을 키운다. 그리고 함수등을 매번 만들면 많은 변수 등이 사용될텐데, 이를 줄임으로써 예기치않은 오류를 최대한 줄일 수 있다는 장점을 가진다.
