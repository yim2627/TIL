# 학습 내용

*  Git, Github
	+ Git으로 짝 프로그래밍을 해보았다. 충돌을 고치고, PR을 올려보는 값진 경험을 해보았고, 차후 더 삽질😁해볼 계획이다.
	+ Git과 Github는 절대 동일한 것이 아니다. Git은 VCS(Version Control System), 로컬에서 관리되는 버전 관리 시스템이고, Github는 저장소를 제공해주는 서버 개념이므로 둘은 각각 다른 역할을 지니고 있다.
	+ Commit은 변경사항을 기록하는 것이다. 또한 변경사항 등을 git log 명령어를 통해 수정 내역 등을 볼 수 있다. 또한 Commit은 폴더 단위로 이루어진다.
	+ Commit의 장점은 어떤 문제사항이 발생하였을 때 문제 발생 전으로 돌아갈 수 있다. 즉 백업의 기능을 지니고 있다.
	+ git add는 왜 쓰는 것인가? - ```staging area```에 다음 변경(commit)을 기록할 때까지의 변경분을 모아놓기 위해 사용한다. </br>```staging area```는 로컬의 작업 디렉토리와 와 원격 저장소의 변경 이력 사이에 다리 역할을 하는데..  커밋할 변경 내용이 원격 저장소에 기록되기 전에 대기하는 임시 저장소라고 생각하면 되는 것 같다. </br>여기서```git add ```명령어를 사용하면 현재 작업 디렉토리에 있는 모든( git add .) 또는 일부(git add [file]) 변경 내용을 스테이징 영역으로 이동시킨다.
	+   편한 GUI를 쓰지않는 이유에 대해 물어봤다. Coda의 답변: 터미널은 모든 컴퓨터에 존재하지만, GUI는 모든 컴퓨터에 있진 않고, 터미널을 사용하여 Git을 사용할 줄 아는 사람은 GUI로도 금방 잘하지만, 반대의 경우는 그렇지않다. (좋은 답변 고마워요 Coda😆)
	+ git pull을 해야 remote branch를 local branch로 가져올 수 있다. -> git checkout -t origin/[remote branch name]
	+ Reset - 특정 시점(과거)으로 돌아가 해당 커밋 아후(과거~현재)의 내역을 모두 삭제하는 명령어
	+ Revert - 특정 사건을 골라서 없던 일로 만든다. 특정 사건을 없애주고, revert 했다는 이력을 남긴다.
	+ Fetch와 Pull의 차이 - 병합을 하냐?(Pull) 안하냐?(Fetch)의 차이 (정말 고마워요 븨븨!👍)
	+ origin/Head란? - 원격 저장소의 상태와 일치하는 로컬 저장소의 상태를 뜻하는 것 같다. (정말 고마워요 븨븨!👍)
	+  git commit --amend:  커밋 메세지 수정 반영 (정말 고마워요 븨븨!👍)
	+  rebase: base를 재배치해준다는 개념이다. 병합할 때도 사용이 되지만, 커밋 메세지를 수정할 때 git rebase -i 명령어로 쓰인다. </br>특정 시점을 잡아 커밋 리스트를 출력하고, vi를 이용하여 reword를 활용하여 커밋 메세지를 변경 한 뒤 continue를 사용하여 rebase 작업을 계속 진행해준다. </br>이 작업은 잡은 커밋의 시점 이후의 커밋들에 영향을 끼치며, 만일 수정하고 싶은 commit이 push된 상태라면 commit을 수정해도 push가 되지 않을것이다. 이때 git push -f 명령어를 이용하여 원격 저장소의 커밋 히스토리를 강제로 덮어씌울 수 있다. </br> 하지만  만약 이전에 push한 커밋들을 다른 개발자들과 공유하고 있었다면 커밋 히스토리의 불일치가 발생해 흔히 말하는 "git이 꼬였다." 하는 상황이 발생할 수 있다 (정말 고마워요 븨븨!👍)
